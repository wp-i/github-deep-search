# V1 交付质量与系统测试推进计划

**版本：** 草案 v0.1
**适用范围：** GitHub Deep Search V1 首次公开交付
**更新日期：** 2026-07-10

## 1. 目的与交付定义

V1 不是“能返回若干仓库”的交付，而是一次可被复查的开源调研：使用者能够知道系统如何理解需求、搜索过什么、每项结论由什么证据支持、哪些能力尚未确认，以及一次失败如何被复现和追踪。

本计划解决三项当前风险：

1. 模糊需求会得到低关联项目或空结果，报告没有决策价值。
2. 当前自动化测试偏重规则和模拟流程，未覆盖足够多的未见用户需求与真实 Provider 组合。
3. 项目缺少能让审查者独立验证的质量资产，源码价值主要体现为规则约束而非可复现的质量证据。

### V1 质量原则

- 产品语义只来自当前用户需求、LLM 生成的 `SearchSpec` 和真实仓库证据。
- 不为评测集、语言、领域或预期仓库加入关键词包、同义词表、翻译回退、白名单、黑名单或特判。
- 报告宁可明确“不确认”，也不得把弱线索写成支持；但有证据的局部匹配和相邻线索不得被无意义清空。
- 对确认的失败，必须追溯 `用户输入 → SearchSpec → 搜索计划 → 仓库证据 → 排序/报告`，在最早错误阶段修复。
- 真实评测保留每次运行的输入、Provider 配置指纹（不含密钥）、阶段追踪、结构化结果和可读报告。

## 2. 当前基线（2026-07-10）

| 维度 | 已有资产 | 主要缺口 |
| --- | --- | --- |
| 自动化测试 | 约 160 个测试，覆盖解析、召回、证据门、报告、Web 和架构规则 | 没有覆盖率采集与阈值；测试数量不能代表需求空间覆盖 |
| 流程可观测性 | `tests/test_pipeline_trace.py` 已按解析、查询、候选、README、证据、分析六阶段校验 | 尚未成为每次真实评测都产出的统一工件 |
| 真实运行 | `tests/test_live_eval.py` 有一个需显式开启的 live 用例；`docs/REAL_RUNS.md` 有一次历史捕获 | 单一真实用例无法说明对模糊、多语言、多约束需求的稳健性；现有 live 用例未执行统一的六阶段产物判定 |
| CI | 合并前运行非 `e2e/live` 测试和 Python 编译 | 没有质量趋势、覆盖率、浏览器发布前检查或真实评测的人工门禁 |
| 链接复核 | `e2e` 会在浏览器中打开本地 Web 页面并渲染固定 API 响应 | 没有从真实报告点击或打开候选项目地址、检查跳转地址和页面可达性的复核 |
| 审查材料 | README、证据门架构说明、截图和架构规则 | 缺少场景集、评分准则、失败登记、可复现运行包和发布结论 |

该基线只描述测试与交付能力，不预判任何具体失败发生在解析、检索、证据或报告阶段。真实缺陷必须按第 6 节的诊断流程归因。

## 3. 测试模型

### 3.1 质量维度

| 维度 | 要回答的问题 | V1 主要证据 |
| --- | --- | --- |
| 需求理解 | 核心目标、硬约束、可选项是否被正确分离？ | `SearchSpec`、解析阶段 trace、人工盲评 |
| 发现能力 | 搜索计划是否覆盖仓库、代码、topic、issue 等合适通道？ | 查询计划、候选来源、候选池统计 |
| 证据真实性 | 报告中的“支持”能否在 README、路径或源码局部证据中复查？ | `evidence_coverage`、证据片段、人工抽查 |
| 排序与呈现 | 高置信、参考、相邻线索是否一致且不误导？ | tier、分数、缺口说明、报告审查 |
| 失败可用性 | 无直接匹配时是否给出可行动的解释和下一步，而非无意义结果？ | 空结果/相邻线索报告、搜索渠道说明 |
| 稳定性与成本 | Provider 波动、限流、异常时是否可诊断，预算是否受控？ | 运行记录、错误分类、耗时和用量 |
| 体验与可访问性 | Web、CLI、MCP 的输出是否可用、可复制、可导出？ | `e2e`、接口契约、人工冒烟清单 |
| 决策体验 | 用户能否在不阅读内部实现的情况下判断可用性、依据、缺口和下一步？ | 30 秒报告审查、摘要/报告层级、用户代理评审 |

### 3.2 测试金字塔

1. **确定性单元与契约测试：** 继续验证结构、去重、证据局部性、评分上限、序列化和错误呈现。合成数据只验证领域中立的不变量，绝不把真实案例的产品语义编码进 fixture。
2. **阶段集成测试：** 对每一阶段记录输入输出和不变量，能指出失败最早出现在解析、搜索计划、候选、证据还是报告。
3. **真实 Provider 评测：** 使用独立、版本化但不被运行时读取的场景卡；每个确认失败至少独立运行两次。
4. **发布体验测试：** 覆盖新用户启动、缺少凭据、网络/限流、结果复制与 JSON 导出，以及窄屏浏览器流程。
5. **人工审查：** 由不了解实现的人依据评分卡审查随机抽取的报告和证据链接。

## 4. 场景集与评测设计

### 4.1 场景分层

场景集由“需求形态 × 约束复杂度 × 语言 × 结果形态”正交组合，而不是按热点产品词堆砌。每个场景使用真实、独立撰写的用户需求，并记录其来源或撰写日期。

| 场景族 | 覆盖目的 | 最低 V1 样本数 |
| --- | --- | ---: |
| 清晰单一能力 | 建立正常路径基线 | 8 |
| 模糊意图 | 验证澄清式解析、合理相邻结果和不确定性表达 | 8 |
| 多步骤工作流 | 验证 LLM 将步骤归纳为可搜索能力，而非逐字硬塞入核心要求 | 6 |
| 复合硬约束 | 验证领域、动作、对象必须由局部组合证据共同支持 | 6 |
| 多语言与混合输入 | 验证当前请求的语义被保留，且不依赖静态翻译补救 | 6 |
| 罕见或无直接匹配需求 | 验证空结果和相邻线索的可用性 | 4 |
| Provider/运行异常 | 验证密钥缺失、限流、超时和报告交付失败的诊断 | 4 |
| **合计** | 首轮交付评测池 | **42** |

每个场景卡至少包含：`case_id`、原始需求、语言、场景族、人工定义的核心结果方向、可选约束、风险级别、评测日期和审查人。它**不得**包含预期仓库名、静态同义词、允许仓库清单、固定报告文本，且不得被运行时代码加载。

### 4.2 双盲评分卡

审查人先看需求与报告，再打开报告中给出的证据；不看系统内部提示词、测试实现和历史评分。每项 0–2 分：

| 项目 | 0 分 | 1 分 | 2 分 |
| --- | --- | --- | --- |
| 需求理解 | 核心方向偏离 | 部分理解但约束混乱 | 核心和可选项分离合理 |
| 候选相关性 | 候选与核心方向无关 | 有局部关联 | 至少一个候选对核心方向有实际价值 |
| 证据可信度 | 支持声明无法复查或夸大 | 有证据但范围不清 | 每个支持声明与局部证据一致 |
| 层级一致性 | 高低置信混淆 | 少量表述不清 | reliable/reference/adjacent 与证据一致 |
| 失败可行动性 | 空白、噪声或无下一步 | 有解释但不具体 | 明确未确认项、已搜渠道与下一步 |

分数用于发现趋势，不以“必须命中某仓库”作为通过条件。若审查人意见不一致，记录分歧和证据，而不是修改实现去迎合单个案例。

### 4.3 真实运行工件

每次真实运行写入一个不被 Git 忽略的、已脱敏的发布评测目录；密钥、完整响应原文、私有路径和个人信息不得入库。建议的最小结构：

```text
docs/evaluations/<release-id>/<case-id>/<run-id>/
  request.json          # 场景卡 ID、原始请求、配置指纹
  trace.json            # 六阶段计数、渠道、耗时、错误分类
  report.json           # 序列化结果，含 evidence_coverage
  report.md             # 可读交付报告
  review.md             # 评分、抽查证据、审查结论
```

`run-id` 应包含 UTC 时间和随机后缀；同一确认失败的回归必须保存至少两份独立运行工件，比较不变量而非项目名和排序。

### 4.4 项目地址的浏览器复核

候选地址是报告的一部分交付物，不能只校验字符串存在。V1 增加两层检查：

1. **确定性地址契约：** 验证序列化后的 URL 使用 HTTPS、指向公开 GitHub 仓库地址，且仓库 owner/name 与报告中的候选身份一致；该检查不访问网络。
2. **真实浏览器复核：** 对每份发布评测报告的全部 reliable 候选和 Top 1 reference/adjacent 候选，在隔离浏览器上下文中从渲染后的报告链接点击或读取真实 `href` 后打开。记录初始 URL、最终 URL、HTTP/导航结果、页面标题、检查时间和失败原因。

复核通过的最低条件是：页面可在允许的网络与认证条件下加载，重定向后的规范仓库地址仍对应报告候选，且不是登录页、404、仓库移除页或无关地址。若 GitHub 限流、网络受限或需要登录，工件必须标记为“环境未验证”，不得伪造为可达。

浏览器复核只证明**链接可达且身份一致**，不证明项目具备所述能力；能力结论仍必须由报告的 README、路径或源码证据支持。为避免把外站波动误判为产品缺陷，链接不可达应先按网络、认证、限流、地址生成四类归因。

### 4.5 非专业、口语化与不完整输入专项

“没有软件基础”的表达不是异常输入，而是 V1 的核心覆盖面。场景池新增至少 8 个“日常任务描述”案例：包含代词、省略主语、业务结果词、输入不完整、未指明平台/数据源，以及把操作步骤与期待结果混在一起的叙述。

例如，用户可能表达为：“我有个需求，我给他链接，他把对应链接在 app 端的价格实现链路明细统计给我，最后输出到手价。”此类场景的评审不要求命中某个预设项目，而检查：

- `SearchSpec` 是否保留“链接输入、价格获取/明细、最终到手价输出”这一当前请求中的核心方向。
- 未说明的商品平台、价格构成、账号/地区、优惠规则和具体 App 是否被诚实保留为未知或待澄清，而不是擅自变成硬约束。
- 搜索计划是否由当前生成的 `SearchSpec` 形成；测试不得要求固定英文翻译、固定同义词或固定查询。
- 若没有直接项目，报告是否提供证据支持的局部/相邻线索，并说明未确认的关键环节和下一步澄清问题。

这些案例以场景卡方式保存在运行时之外，由不同撰写者提供原话和人工评审结论；它们不得进入解析提示词示例、静态清理逻辑、关键词表、仓库名单或固定期望报告。对同一需求可收集独立的口语变体来衡量稳健性，但只比较结构完整性、证据真实性和可行动性，不比较预期仓库名。

### 4.6 真实运行的全链路产物判定

现有 `test_pipeline_trace.py` 可以在模拟 `SearchSpec`、GitHub 与分析结果的条件下检查六个阶段；它不能替代真实运行。`test_live_eval.py` 当前也只对基本结构做断言，尚未调用统一的六阶段判定。

每次真实评测必须执行一个与运行时代码解耦的 `live trace contract`，并把结果写入第 4.3 节的 `trace.json`。它覆盖从输入到交付的每项产物：

| 阶段 | 必须判定的产物 | 允许的失败状态 |
| --- | --- | --- |
| 输入与解析 | 原始输入、非空 intent、核心/可选项、查询计划、`evidence_aliases` 的结构完整性 | Provider/解析失败须有分类和可读报告 |
| 搜索执行 | 已计划与实际执行的渠道、查询计数、预算/限流告警 | 预算或网络中断须说明未执行的渠道 |
| 候选与采集 | 候选数、来源、README/路径/源码采集结果 | 零候选须保留搜索范围和失败原因 |
| 证据与分层 | 每个返回项目的 URL、`evidence_coverage`、支持/未知/差异状态、tier 和分数 | 未确认能力只能进入 reference/adjacent 或被明确排除 |
| 报告与导出 | Markdown、JSON、Web 渲染、复制/下载字段、用量、警告和可读空结果说明 | 写入/编码/渲染失败一律视为本次端到端失败 |
| 外部链接 | 第 4.4 节的 URL 契约和浏览器复核记录 | 网络限制只能标记环境失败，不能标记通过 |

成功不等于“返回了项目”。一条真实运行只有在所有必需产物可读、可关联且没有触发第 5.1 节的不变量时才算完成；没有直接匹配但给出完整 trace、证据边界和可行动下一步的运行可以通过。每次失败必须保留最早失败阶段，供第 6 节归因。

### 4.7 多 Agent 对抗评测

除人工双盲审查外，V1 使用相互独立的 Agent 角色主动寻找反例。它们评审工件，不参与运行时决策，也不得把发现的特定产品词、仓库名或提示词写回搜索代码。

| 角色 | 只看什么 | 要攻击的问题 | 有效输出 |
| --- | --- | --- | --- |
| 用户代理 | 原始需求与最终报告 | 用户能否在 30 秒内知道能否使用、为何可信、差什么、下一步做什么 | 决策缺口及对应报告段落 |
| 语义红队 | 原始需求、`SearchSpec`、实际查询 | 系统是否遗漏核心目标、把未知项擅自硬化，或把可选项抬成核心 | 最早错误阶段与结构化字段差异 |
| 证据检察官 | 报告主张、`evidence_coverage`、仓库证据和链接复核 | 是否有支持声明无法回到局部证据，或弱证据被提升为可靠 | 主张 → 缺失/冲突证据的可复核链 |
| 架构审查员 | 变更 diff、模块依赖、ADR、契约测试 | 是否引入运行时/评测泄漏、静态语义补丁、补偿分支或无行为抽象 | 文件/依赖/测试引用和违反的契约 |
| 运行可靠性审查员 | 两次 run trace、用量和错误记录 | 是否把 Provider、网络、限流或报告交付故障误报为成功/产品无匹配 | 失败分类、工件完整性和差异说明 |

**执行协议：**

1. 场景和对抗提示由与实现者隔离的人员或 Agent 产生；运行时不读取这些材料。
2. 每个 Agent 只获得完成职责所需的最小工件，避免因看到测试实现或其他 Agent 的结论产生锚定。
3. 至少对 30% V1 场景执行用户代理、语义红队和证据检察官三种审查；P0/P1 修复再加入架构与可靠性审查。
4. Agent 输出写入 `adversarial-review.json`：角色、工件版本、主张、证据引用、严重等级、反例和不确定性。不同意见必须并列记录。
5. 只有能关联源码、trace、报告或仓库证据的结论才能创建缺陷；缺陷仍按第 6 节定位最早阶段并做两次真实回归。

对抗评测的目标是提高发现盲区的能力，不是让 Agent 投票选“正确仓库”。任何 Agent 的建议都不能成为新的静态别名、翻译词、repo allowlist、固定报告或样例特判。

V1 默认通过 `scripts/run_adversarial_review.py` 使用当前配置的 OpenAI-compatible API 为每个角色创建独立评审上下文。该脚本只读取已导出的 JSON 报告与显式提供的源码上下文，不调用搜索运行时；如需模型多样性，使用不同 Provider 配置独立重跑并保存各自工件。

## 5. 发布门禁与量化指标

### 5.1 不变量门禁（零容忍）

以下任一项失败即阻断 V1：

- 报告把未确认的核心能力标记为支持或可靠匹配。
- 复合要求仅因分散关键词、截图路径、平台/输出等弱证据被确认。
- 有意义的、证据支持的部分匹配仍存在，但报告无解释地返回空项目列表。
- 真实回归运行未能完整、可读地写出报告，或报告/证据工件缺失。
- 运行时或评测实现引入产品词包、同义词/翻译回退、仓库名单、提示词特判或 fixture 形状补丁。
- 密钥、令牌、私有仓库内容或个人数据进入提交的评测工件。

### 5.2 建议的 V1 量化门槛

首轮 42 个场景完成后冻结基线；以下阈值作为发布评审的建议，任何调整必须记录理由与批准人。

| 指标 | 计算方式 | 建议门槛 |
| --- | --- | ---: |
| 结构完整率 | 真实运行中生成有效 `SearchSpec`、阶段 trace、报告的比例 | ≥ 95% |
| 证据声明准确率 | 人工抽查中“支持”声明可由本地证据复查的比例 | 100% |
| 链接身份与可达率 | 第 4.4 节范围内通过真实浏览器复核的报告项目链接比例 | 100% 或明确环境失败 |
| 全链路产物完整率 | 真实运行通过第 4.6 节所有必需产物判定的比例 | ≥ 95% |
| 决策摘要可用率 | 用户代理确认报告明确回答可用性、依据、缺口和下一步的比例 | ≥ 85% |
| 对抗审查可归因率 | 有证据的 Agent 问题可定位到最早阶段或被证伪的比例 | 100% |
| 核心相关可用率 | 评分卡“候选相关性”≥1 且“失败可行动性”≥1 的比例 | ≥ 85% |
| 报告可用率 | 评分总分 ≥ 7/10 的比例 | ≥ 80% |
| 严重缺陷 | P0/P1 未关闭数量 | 0 |
| 双运行不变量通过率 | 确认失败修复后，两次运行均未触发零容忍项的比例 | 100% |
| 自动化回归 | CI 中确定性测试、编译、发布前 e2e 的通过率 | 100% |

代码覆盖率在 V1 只作为趋势指标：先用 `pytest-cov` 建立可复现基线，再按解析、引擎、Provider、Web 分模块发布趋势。不得为了提高百分比添加无行为断言或样例形状测试；未覆盖的风险路径必须转成场景或缺陷登记。

## 6. 缺陷诊断与回归协议

任何“不符合用户预期”的报告，先创建缺陷记录，再决定是否改代码。记录必须按以下顺序完成：

1. 固化原始用户输入、运行时间、配置指纹、Provider 响应状态和完整工件。
2. 检查 `SearchSpec`：核心能力、可选项、查询与 `evidence_aliases` 是否仍代表当前需求。
3. 检查搜索计划和候选来源：正确的查询/渠道是否实际执行，预算是否在证据阶段耗尽。
4. 检查仓库证据：README、路径、源码证据是否被采集，是否局部支持复合要求。
5. 检查证据门、tier、分数和报告：是否将 unknown/missing/different 正确呈现。
6. 将缺陷归到**最早错误阶段**，提出最小、领域中立的修复；拒绝任何下游补偿或测试专用重写。
7. 对确认失败，在真实配置的 Provider 上独立运行至少两次，并保存两份完整报告和 trace；比较“无虚构证据、tier 不矛盾、有意义相邻结果不丢失”等不变量。

### 缺陷优先级

| 级别 | 定义 | 发布处理 |
| --- | --- | --- |
| P0 | 虚构能力、泄露秘密、无法生成/交付报告 | 阻断发布，立即修复 |
| P1 | 核心需求被明显错解、相邻结果误报为可靠、常见流程不可用 | 阻断发布，修复并双运行回归 |
| P2 | 排序、解释或交互明显降低决策效率，但结论仍可证据复查 | 可在 V1 前修复；否则记录为已知限制 |
| P3 | 文案、轻微布局、非关键可观测性缺口 | 纳入后续迭代 |

### 测试工程变更的真实验证

凡是修改测试代码、评测 harness、场景卡、trace schema、fixture、浏览器检查、测试脚本或测试报告交付，完成后必须用真实配置的 Provider 执行至少一个端到端案例。单元测试、mock、fixture 或“测试自身通过”均不能替代这次验证。

真实运行必须同时证明两件事：

1. 新增/调整的测试机制能从真实产物中正确记录并验证预期字段、阶段或报告行为。
2. 它能暴露旧测试未验证的盲区，例如缺少阶段工件、无证据的支持声明、tier 矛盾、报告交付不完整或其他真实未观测条件。

评测工件必须记录原始请求、配置指纹、trace、可读报告、旧盲区、当前检测结论和后续动作。不能用固定仓库、样例词、静态别名或测试专用报告伪造第二项证明；若真实运行无法证明两项，测试工程仍视为未完成，必须继续改进后重新验证。

### 回归闭环退出条件

真实验证不是最终勾选项，而是闭环。每次真实运行、浏览器复核或独立 Agent 审查发现可复核问题时，必须：定位最早错误阶段 → 最小合规修复 → 跑受影响的确定性测试 → 重跑**同一真实案例** → 检查原问题已经解决且没有虚构证据、tier 矛盾或有意义相邻结果丢失。

最新真实运行若仍包含未关闭的、证据支持的问题，本轮测试、修复和发布候选均不得宣告完成。重跑时发现的新问题立即成为新的待解决项，并从最早错误阶段重新进入同一循环；LLM 辅助搜索的最终成功迭代仍需两次独立真实运行。

## 7. 分阶段推进

### Phase 0：冻结基线与可追溯性（第 1 周）

**目标：** 将现有测试、真实运行和未提交改动隔离，建立统一评测工件格式。

- 建立场景卡模板、评分卡模板、缺陷模板和工件目录约定。
- 为现有 six-stage trace 输出定义稳定 JSON schema；记录 Provider 配置指纹和用量，不记录密钥。
- 落地第 4.6 节的 `live trace contract`，使真实运行从输入到报告、导出和链接复核都有机器可读结论。
- 将第 4.4 节的 URL 契约和浏览器复核作为发布评测工具，而不是把外站页面内容写入 unit fixture。
- 用至少 8 个非专业口语输入进行试点；其中必须包含链接输入、未说明数据源/平台和业务结果混合表达。
- 在 CI 中增加 JUnit 结果与覆盖率采集，仅发布趋势，不设虚假百分比目标。
- 清点当前 `live`、`e2e`、单元测试各自的边界，明确其不替代的风险。

**完成证据：** 基线清单、42 个场景卡初稿、一次端到端工件样例、CI 产物说明。

### Phase 1：场景评测与故障归因（第 2–3 周）

**目标：** 发现真实需求空间里的最早失败阶段，而不是堆叠新规则。

- 使用两个独立 Provider 或同一 Provider 的独立运行执行 42 个场景。
- 每个场景至少保留一次完整工件；P0/P1 和准备修复的案例运行两次。
- 两名审查人对至少 30% 场景双盲评分，记录不一致项。
- 对至少 30% 场景运行第 4.7 节的用户代理、语义红队和证据检察官；将可复核反例纳入缺陷矩阵。
- 生成缺陷矩阵：场景族 × 最早失败阶段 × 严重等级 × 复现链接。

**完成证据：** 初始质量分数、失败 Pareto、每个 P0/P1 的阶段归因，且没有产品词表式修复提案。

### Phase 2：根因修复与分层回归（第 3–5 周）

**目标：** 逐项修复 P0/P1，并把每个修复转化为领域中立的不变量测试。

- 只修改最早错误阶段：解析、证据采集、通用验证或排序/报告中的一个明确位置。
- 每个修复增加最小的结构/集成测试；真实场景仅验证工件和不变量，不固定项目名或排序。
- 对每个确认失败执行两次真实 Provider 回归，检查报告可读与工件完整。
- 在每次迭代后重跑受影响的场景族，追踪质量分数是否回退。

**完成证据：** 所有 P0/P1 关闭；两次回归工件；变更说明包含完整阶段追踪和架构规则自检。

### Phase 3：交付候选与审查包（第 5–6 周）

**目标：** 将质量结论变成外部审查者可独立验证的发布资产。

- 运行全量确定性测试、编译、Web `e2e`，并在干净环境执行一键启动冒烟测试。
- 重新执行冻结的关键场景子集；发布评测汇总不展示密钥或无依据的“成功率”。
- 完成代码审查、依赖/许可证检查、文档链接检查和已知限制审查。
- 形成 V1 发布决议：通过、附条件通过或延期；任何豁免必须关联缺陷、风险和恢复计划。

**完成证据：** 第 8 节全部工件、门禁结果、已知限制和可复现命令。

## 8. 面向开源审查的 V1 质量包

除 `AGENTS.md` 的架构约束外，交付应让审查者看到“如何证明系统值得信任”：

0. **工程治理：** [AI 辅助工程治理与源码审查提升方案](AI_ASSISTED_ENGINEERING_GOVERNANCE.md) 将语义所有权、证据溯源、阶段失败、变更证明和评测隔离落实为可执行契约。
1. **质量声明：** 本文、`docs/evidence-gating.md` 和 README 中的边界、指标、已知限制保持一致。
2. **可复现评测包：** 发布场景卡、脱敏 trace、报告、评分和汇总；任何人可用自己的凭据重跑。
3. **阶段可观测性：** 每份评测报告能定位到六阶段的输入输出计数和候选来源，不只是最终分数。
4. **证据可审计性：** 报告的确认、缺口、相邻线索均能回到 README/路径/源码片段，不以模型断言替代证据。
5. **质量趋势：** CI 测试、覆盖率趋势、真实评测指标和缺陷关闭率按 release 比较，而非单次截图宣传。
6. **诚实的限制：** 说明 Provider 非确定性、速率限制、成本和网络前提，并公开未解决的 P2/P3 风险。

建议在 V1 发布说明中附上一页“质量摘要”：场景总数与分布、双盲样本量、各项门禁结果、两次回归的比对结论、P2/P3 已知限制、评测工件位置和重跑命令。

## 10. 首次执行顺序

1. 指定质量负责人、评测审查人和发布决策人。
2. 先完成 Phase 0 的模板与基线，不修改搜索运行时。
3. 以 8 个试点场景校验工件格式、评分一致性和成本，再扩展至 42 个场景。
4. 依据 Phase 1 的最早失败阶段排序修复工作，不按单个案例的“想要仓库”排序。
5. 完成所有 P0/P1 双运行回归后，才进入 Phase 3 发布候选。

这条顺序的核心是：先让失败可观察、可复现、可归因，再提升解析、发现和证据质量；绝不通过静态语义补丁把一次评测“做绿”。
