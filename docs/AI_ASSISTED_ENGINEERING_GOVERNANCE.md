# AI 辅助工程治理与源码审查提升方案

**状态：** V1 交付前实施路线
**目标：** 让克隆仓库的审查者能从源码、测试和发布工件中验证工程判断，而不只看到一份“禁止 AI 抄近道”的规则文本。

## 1. 问题定义

`AGENTS.md` 已经建立了重要底线：不允许静态语义补丁、样例特判、下游补偿和不可复现的修复。这是必要条件，但不是工程价值的充分条件。

高质量 AI 辅助开发的价值不在于“写了很多限制”，而在于把不确定的模型行为放进可验证的系统：

- **语义有所有权：** 当前请求的产品含义只能由解析器产出，并能被追溯。
- **结论有出处：** 每个“支持”判断都能回到仓库局部证据，而非模型自信。
- **失败能定位：** 一次不佳结果能定位到最早失败阶段，而非继续添加补丁。
- **变更能证明：** 每次修改都说明改了哪个契约、用什么不变量和真实运行证明没有回退。
- **交付能复核：** 外部审查者不需要相信作者；他们能用自己的凭据和流程复核质量结论。

本方案的原则是：**增加可执行的结构和证据，而不是再增加一层口号或把架构过度抽象化。**

## 2. 审查者应在 15 分钟内看见什么

一个新审查者应能沿着下列路径理解项目，而不用先读完整个 `engine.py`：

1. `docs/ARCHITECTURE.md`：系统边界、数据流、模块职责、依赖方向和失败语义。
2. `docs/adr/`：关键取舍为何存在，例如“语义归 LLM、验证归确定性代码”“不把真实评测数据接入运行时”。
3. `github_deep_search/models.py`：稳定的数据契约，包括需求、候选、证据、分析、追踪和失败状态。
4. 对应模块与契约测试：一次结论从 `SearchSpec` 到 `EvidenceCoverage` 再到报告的实现和不变量。
5. `docs/evaluations/<release-id>/` 与质量摘要：真实 Provider 运行、两次回归比较、链接复核和已知限制。

这条路线不是额外的文档负担；每一层都必须由代码、测试或发布工件实际支撑。没有可执行或可复核载体的“最佳实践”不计入交付价值。

## 3. 五个可执行工程契约

### 3.1 语义所有权契约

**责任划分：** `SearchSpecParser` 解释当前用户请求；后续模块只能消费结构化的 `SearchSpec`/`Requirement`，不能重新以静态业务规则解释自然语言。

**代码形态：**

- 解析器输出完整的 `intent`、核心/可选能力、频道查询、`evidence_aliases` 和组件分组。
- 搜索、证据、排序模块只读取当前 `Requirement` 的字段；通用代码只做结构校验、规范化、去重、局部性和覆盖度计算。
- 无效解析结果必须带有机器可读的原因和回退来源，不能静默退回到一串“看似合理”的固定查询。

**可执行检查：**

- 结构契约测试验证字段完整、关键映射覆盖和输入/输出不变式。
- AST/模块边界检查验证运行时不读取 `tests/`、`docs/evaluations/`、`tmp/` 或示例数据目录。
- 真实评测将原始输入、`SearchSpec` 和实际执行查询一并保存，以便审查者确认没有隐藏的语义扩展。

### 3.2 证据溯源契约

**责任划分：** LLM 可以提出解释，确定性证据门决定哪些主张能被呈现为支持。

**代码形态：**

- 将证据从当前的展示字符串逐步演进为带 `source_kind`、仓库路径/README 定位、局部片段、特征关联和采集时间的结构化引用。
- `ProjectAnalysis` 的每项已支持能力都关联至少一个 `EvidenceReference`；`unknown`、`missing`、`different` 保持不同状态，不能相互伪装。
- 报告构建器只渲染经证据门批准的主张；它不重新计算能力匹配，也不改写未知项为缺失项。

**可执行检查：**

- 契约测试检查“支持主张 → 证据引用 → 仓库局部位置”的完整链。
- 复合要求测试检查领域、动作、对象在同一局部窗口共现，而非跨 README 拼接。
- 发布评测随机打开报告中的证据和项目链接，记录可达性与身份一致性；链接可达不被视为能力证明。

### 3.3 阶段边界与失败物化契约

**责任划分：** 编排器负责顺序、预算和资源关闭；各阶段对自己的输入、输出和失败状态负责。

**目标模块边界（按现有行为渐进抽取，不做空洞分层）：**

```text
入口（CLI / Web / MCP）
        │
        ▼
PipelineOrchestrator
  ├─ RequirementInterpreter   解析与结构校验
  ├─ DiscoveryPlanner         频道计划与候选发现
  ├─ EvidenceCollector        README、路径、源码采集与覆盖
  ├─ CandidateEvaluator       证据门、tier、分数与差异
  ├─ ReportBuilder            Markdown/JSON/Web 投影
  └─ RunTracer                阶段事件、用量、失败与工件
        │
        ▼
Provider adapters（GitHub / LLM / Tavily）
```

`PipelineOrchestrator` 不应保留领域判断；Provider adapter 不得导入 Web、CLI、测试场景或报告策略。拆分只在下列条件同时满足时进行：存在稳定的数据输入/输出、可以写出契约测试、并能消除当前 `engine.py` 中一个实际的职责混合点。

**失败物化：** 引入有限的阶段/原因分类（例如 `parse`、`discovery`、`evidence`、`analysis`、`report_delivery`，以及 `provider_auth`、`rate_limit`、`network`、`invalid_output`）。用户报告保留可读解释；机器追踪保留阶段、原因、是否可重试和未完成工件。不要用模糊 warning 字符串作为唯一诊断接口。

**可执行检查：**

- 每个阶段输入/输出的 schema 测试和失败状态测试。
- 依赖方向测试：入口可依赖编排器；编排器可依赖领域模块和 Provider 协议；Provider 不反向依赖入口/报告；运行时不依赖评测资产。
- 对实际 Provider 运行执行六阶段 `live trace contract`，即使零候选或网络失败也必须产生完整、可读的追踪结论。

### 3.4 交付投影契约

同一 `SearchReport` 是 CLI、Web、MCP、JSON 和 Markdown 的唯一事实来源。各入口只做协议/渲染适配，不能自行补充候选、分数、结论或测试数据。

**可执行检查：**

- 序列化 schema 对每个入口共享；Web/CLI/MCP 针对同一报告比较相同的候选身份、tier、分数、证据状态和用量。
- URL 在序列化、渲染和浏览器打开之间保持同一仓库身份。
- 报告写入、编码、HTML 渲染、复制与下载任一步失败都属于端到端交付失败，不能因“引擎已经找到候选”而标记成功。

### 3.5 变更证明契约

每个涉及解析、搜索、证据、排序、报告或使用量的变更都提供一个小型“变更证明”，而不是只附一句“已修复”。证明至少包括：

| 项目 | 内容 |
| --- | --- |
| 变更类别 | 解析、证据采集、通用验证、排序/报告、UI 或使用量 |
| 流程定位 | 用户输入到报告的阶段追踪和最早错误阶段 |
| 契约影响 | 新增/修改了什么输入、输出或不变量 |
| 反方案 | 为什么没有使用词表、同义词、翻译回退、特判或下游补偿 |
| 验证 | 确定性测试、受影响评测场景和真实双运行回归（如适用） |
| 风险 | Provider 波动、成本、兼容性和已知未覆盖边界 |

变更证明可以放在 PR 描述和关联 ADR，不应在运行时保留，更不能变成自动注入提示词的样例库。

### 3.6 双受众与独立 Agent 审查契约

项目有两类必须分别满足的受众，不能以其中一方的“通过”代替另一方：

| 受众 | 他们需要快速判断什么 | 系统必须给出的可复核答案 |
| --- | --- | --- |
| 使用报告的用户 | 能不能用、为什么、差什么、下一步该做什么 | 候选 tier、局部证据、未确认项、改造成本/风险、下一步验证动作 |
| 审查源码的 Agent/开发者 | 结论如何产生、边界是否真实、修改是否会污染语义 | 数据契约、依赖方向、trace、结构化证据、ADR、契约测试与真实回归工件 |

用户报告的首屏/摘要必须支持“30 秒决策”：先给出直接可用、参考或相邻线索的结论，再说明最强证据、核心缺口与一个明确下一步。信息密度不等于把内部 token、提示词、无关搜索细节堆给用户；这些细节只应出现在可展开的 trace 或导出工件中。

独立审查 Agent 的任务不是根据个人偏好重写实现，而是对照本文件的契约提出可复核反例：哪一项主张没有证据、哪一条依赖方向被破坏、哪个失败未被物化、哪个测试把场景语义教回了运行时。所有问题必须引用文件、trace 或报告证据；无法落到工件的意见只能作为探索线索，不能直接成为缺陷结论。

## 4. 从“规则”到“可见能力”的仓库资产

| 资产 | 审查价值 | 实施方式 | 验收方式 |
| --- | --- | --- | --- |
| `docs/ARCHITECTURE.md` | 一页看清数据流、职责和依赖方向 | 基于现有模块写明真实边界，不画愿景图 | 每个公开模块与文档职责相符；代码审查更新 |
| `docs/adr/NNNN-*.md` | 展示关键取舍和替代方案 | 从 3–5 个高风险决策开始 | 每份 ADR 有状态、上下文、决定、后果和验证链接 |
| `RunTrace`/失败模型 | 把模型/Provider 不确定性变成可审计数据 | 在领域模型中定义稳定 schema，编排器逐阶段填写 | `live trace contract`、JSON schema、脱敏工件 |
| 结构化证据引用 | 让“为什么推荐”可以沿源码回查 | 从 `EvidenceCoverage` 扩展，先不重写评分算法 | 每个 supported 主张有可渲染引用 |
| 架构契约测试 | 将边界从文字变为 CI 门禁 | AST 依赖检查、schema 检查、投影一致性检查 | 规则变更必须连同测试和 ADR 更新 |
| `scripts/verify_release.py` | 用一个命令生成发布证据清单 | 汇总测试、trace schema、链接复核、工件哈希与脱敏检查 | 生成可提交的 release manifest，不访问或写入运行时数据 |
| 评测与质量摘要 | 让公开质量主张可复现 | 沿用 `V1_DELIVERY_TEST_PLAN.md` 的目录和评分卡 | 发布说明链接到场景、运行与已知限制 |
| PR/Issue 模板升级 | 让协作持续遵守同一工程方法 | 增加变更类别、最早阶段、契约、风险和验证字段 | 解析/证据类 PR 缺字段即不合并 |

这些资产的共同特征是：审查者能看到**边界、数据、证据、失败和取舍**，而非只看到一个大型“智能搜索”函数和很多样例断言。

## 5. 实施顺序与最小纵切片

不要同时进行大型重构、测试扩充和文档美化。第一条纵切片必须从一个完整但小的可审查能力开始。

### Slice A：可观测运行，不改搜索策略

1. 新建 `docs/ARCHITECTURE.md` 和首批 ADR：语义所有权、证据门、评测资产隔离。
2. 在 `models.py` 定义最小 `RunTrace`、`StageOutcome`、`FailureKind` 和结构化证据引用；保留现有公共 JSON 字段的兼容投影。
3. 让 `DeepSearchEngine` 在现有阶段填充 trace，不改变搜索查询、证据评分或排序决策。
4. 为 trace/失败模型加入 schema、序列化和入口投影一致性测试。
5. 用一个真实场景产出脱敏 trace、Markdown、JSON 和 release manifest。

**完成定义：** 审查者可以看到一次结果为什么结束、哪些阶段成功、哪些未执行、用了多少资源，以及报告的主张指向什么证据；没有任何产品领域词被新增到运行时。

**当前实现：**

- `github_deep_search/decision_brief.py` 从现有 `Requirement`、证据状态和候选分析生成结构化 `decisionBrief`；Web 将它作为可选决策卡呈现，Markdown 仍保持精简且不暴露未知项。
- `github_deep_search/run_trace.py` 从已完成的 `SearchReport` 建立稳定的五阶段 `runTrace` 投影，供 JSON 导出、评测工件和审查 Agent 使用。
- `github_deep_search/adversarial_review.py` 只消费已导出的报告和可选源码上下文，定义独立角色与可复核 finding schema；它不参与搜索、排序或报告结论。

对抗审查默认复用当前 `LLM_API_KEY`、`LLM_BASE_URL` 与 `LLM_MODEL`，但每个角色在独立上下文中运行。这样 V1 不增加新的密钥面；需要模型多样性时，使用不同 Provider 配置独立执行同一命令并比较工件，而不是把第二模型写死在代码中。

```powershell
python -m github_deep_search "your requirement" --format json > report.json
python scripts/run_adversarial_review.py report.json --output adversarial-review.json
python scripts/run_adversarial_review.py report.json --roles architecture --source-context change.diff --output architecture-review.json
```

对抗输出只是一份审查工件。只有其中引用报告字段、trace、源码或仓库证据的 finding 才能进入缺陷流程；它不能直接触发运行时策略修改。

### Slice B：提取第一个真实职责边界

1. 根据 Slice A trace 找出 `engine.py` 中耦合最强且有稳定输入/输出的职责（优先证据采集或报告投影，而不是凭直觉拆文件）。
2. 抽取一个模块和 Protocol/数据契约，保持行为不变。
3. 增加依赖方向测试、模块契约测试和受影响的真实回归。
4. 用 ADR 记录为何该边界值得存在、替代方案和不做全面重构的原因。

**完成定义：** 模块减少一个真实的职责混合点，测试不依赖内部实现细节，入口行为与发布工件保持一致。

### Slice C：发布证明自动化

1. 完成 `scripts/verify_release.py` 的离线校验部分：schema、工件完整性、哈希、敏感信息模式和链接复核记录存在性。
2. 将确定性检查接入 CI；真实 Provider 和浏览器复核保持显式发布门禁，避免把成本/网络波动伪装为普通单测。
3. 为一次预发布生成质量摘要和已知限制，邀请外部审查者按第 2 节路径复核。

**完成定义：** 发布不是“截图加口头说明”，而是一个可复跑、可审计、可解释的证据包。

## 6. ADR 起始清单

| ADR | 决策 | 审查问题 |
| --- | --- | --- |
| 0001 | 当前请求语义的唯一来源是 `SearchSpec` | 为什么通用代码不能补充产品词义？ |
| 0002 | 证据状态与报告主张分离 | 如何防止模型解释绕过证据门？ |
| 0003 | 真实评测资产与运行时隔离 | 如何避免评测用例变成隐藏提示词/白名单？ |
| 0004 | `RunTrace` 是公共可审计契约 | 如何区分产品无匹配、Provider 失败和报告交付失败？ |
| 0005 | 以渐进职责提取替代“大重构” | 怎样判断一个新模块是真边界而非包装层？ |

## 7. 不应做的“伪高级”工作

- 不为了“架构感”引入空接口、过多目录、事件总线或依赖注入容器。
- 不把规则复制到多个文档、正则扫描表和提示词模板中，造成维护噪声和新的静态补丁面。
- 不把真实场景、热门仓库、截图或评测结论写进 runtime、system prompt 或 golden report。
- 不以代码覆盖率、类数量、ADR 数量或截图数量作为质量本身。
- 不把 Provider 波动、网络失败和零候选混为“模型不行”，也不把有输出混为“用户目标已达成”。
- 不在没有 trace 和契约的前提下重写 `engine.py`；那只会把复杂度移动到更多文件。

## 8. 与 V1 测试计划的关系

`docs/V1_DELIVERY_TEST_PLAN.md` 负责“如何证明交付质量”；本文负责“如何让源码本身持续生产这些证明”。二者共同形成闭环：

```text
架构契约 → 可观测运行 → 场景评测 → 缺陷最早归因
     ↑                                      ↓
ADR/变更证明 ← 领域中立修复 ← 双运行真实回归
```

V1 的首要优先级是 Slice A，而不是增加更多禁止条款。完成 Slice A 后，再以真实 trace 揭示的耦合点决定 Slice B 的模块提取顺序。

## 9. V1 源码审查验收清单

- [ ] 审查者可在 15 分钟内从架构文档、ADR、模型和一个 release 工件理解数据流与失败语义。
- [ ] 语义、证据、阶段失败、报告投影和运行时/评测隔离均有明确所有权和至少一个可执行契约测试。
- [ ] 每个公开的“支持”结论能够关联结构化证据；每个未确认结论保持未知/差异/缺失语义。
- [ ] 真实运行的 trace 不依赖测试 fixture，并能区分业务无匹配、Provider 问题和交付失败。
- [ ] 解析、搜索、证据、排序/报告和使用量变更均有变更证明；确认失败保留两次真实回归。
- [ ] CI 检查架构边界和稳定 schema；真实 Provider、浏览器复核和成本敏感流程作为显式发布门禁。
- [ ] 文档、ADR、测试和 release manifest 相互链接，且没有运行时读取评测资产的路径。
- [ ] 用户报告通过“30 秒决策”审查；独立 Agent 审查的每项结论均链接到源码、trace 或报告证据。
- [ ] 没有为展示“vibe coding”而引入无行为抽象、样例驱动规则或无法验证的治理文本。
